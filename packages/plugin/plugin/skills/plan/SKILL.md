# Plan

Only after this skill is fully loaded, its first user-visible lines must be:

**PLAN MODE ACTIVATED**
I detect planning intent: [reason]. [action].

Apply Root gate: delegate discoverable facts; ask material decisions through `request_user_input`; state safe reversible defaults. Ask one to three blockers: exclusive options, recommended first with impact, no timeout; never repeat or ask discoverable facts. Explorer is mandatory before a second separable repository read/search or any multi-file or symbol fact pass. Librarian is mandatory before a second external source or multi-source, version, or date research. Two lanes; `fork_turns="none"`; specialists never delegate. No reviewer, evidence dir, commit ritual, ceremony.

## Required sequence

1. Load `plan`; inspect task and repo.
2. For qualifying UI or frontend new app, redesign, or visually driven implementation, when Build Web Apps is available, route to `frontend-app-builder`. Before implementation-detail drafting or complete initial plan, follow its complete concept-generation and design-approval workflow. Treat design approval as separate gate, not implementation-plan approval. If unavailable, tell user to enable Build Web Apps through Codex; do not block planning solely on absence. Read-only UI audits bypass this gate: no concept, design approval, or implementation unless separately authorized.
3. Then draft implementation details only after that design approval when the gate applies. Write the complete initial plan: each step names surface, change, outcome, proof; mark material risks and decisions.
4. Only after the initial plan exists, load `plan-review`. Never preload, parallelize, delegate, or imply review.
5. Use `plan-review` once to revise it; keep phases distinct.
6. Present the reviewed executable plan and ask for approval. Do not implement before approval.
7. After approval, implement the approved plan.

Preserve architecture/scope. Stop after approval; no repeated review.
