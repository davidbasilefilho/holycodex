# OpenAI app-builder reference

This project-original, non-verbatim adaptation governs routed work. It has priority over anti-slop, UI/UX, and other frontend references when they conflict. Explicit user, product, accessibility, visible-content, correctness, and performance requirements remain mandatory unless this reference explicitly controls that conflict.

## Standard and route

- Own visual concepting and faithful implementation for new frontend apps, dashboards, games, creative or visually driven sites, hero sections, and requested redesign, restyle, or modernization.
- First design as a senior product designer; then implement the approved spec as an engineer. Image Gen comes first unless the user opts out or this is a small in-system UI fix.
- Make enough clean, airy, readable, distinctive, section-specific concept imagery. Finish only at agency-signoff fidelity: visual, responsive, interaction, asset, and type drift must be repaired, not excused.
- Design the entire requested surface before code. A full page/app/dashboard/game needs more than a header or hero. For long sites generate one fresh, readable concept per major section (normally one to ten); use an optional overview only for rhythm. Generate standalone detail/state concepts when full-screen text, buttons, cards, type, spacing, or color cannot be extracted. Never use a crop or zoom of a compressed overview as the primary reference.
- For compact apps, dashboards, games, and product surfaces, generate the main screen plus needed responsive, state, and asset concepts. Dense areas need separate table, sidebar, inspector, modal, toolbar, chart, form, card, pricing, testimonial, or media concepts.
- Reject or regenerate header-only, cluttered, generic, repetitive, unreadable, over-decorated, impractical, or off-brief concepts; do not add unrequested/reference-absent hero eyebrows, kickers, badges, or pills.
- For a new complex app UI, default to React + Vite unless the user chooses another framework, the repository dictates one, or delivery is explicitly static/single-file.
- Prohibit hero eyebrow, kicker, pretitle, badge, and pill labels unless user-requested or already present in an approved/reference design.

## Brief, concept, and approval

- Follow installed `imagegen` guidance. Brief with exact purpose, audience, sections/states, workflow, supplied copy/nav/CTAs/data/media, responsive needs, and implementation constraints.
- Request code-native UI text and controls; complete branded raster/product/background assets where appropriate; separable assets; reusable component families; an intentional container model; practical HTML/CSS/components; no default card grid or invented hero labels.
- Preserve supplied IA, screenshots, and product story. Do not invent sections, metrics, claims, dashboards, navigation, or a different product story. Name every known/implied section or state requiring a concept.
- Keep a shared palette, type mood, component family, asset treatment, density, and section order across concept details. For games, separately generate transparent sprites/states or sheets, tiles/platforms, collectibles, hazards, goal/checkpoints, props, and two or three depth layers; HUD, score, controls, physics, and collision stay code-native.
- Choose a coherent direction before generation: theme, background character, type character, hero/primary-screen structure, page/app rhythm, two to four component motifs, and one or two motion cues. Default to tasteful, buildable roughly 7/10 creativity, low-to-medium density, generous space, high type discipline, and image-led moments when useful. Clean means edited and legible, not barren.
- Use a clear focal point, short readable headline/task, restrained support, visible primary action, and laptop-scale negative space. Keep headers to brand, essential navigation, and one primary action unless workflow requires more. Prefer one or two strong images/illustrations; icons only when meaningful.
- Vary long-page density, alignment, scale, media/text ratio, and tempo; connect sections through the approved gutters, bands, alignment, type, media frames, color rhythm, and small spacing shifts. Do not invent large component families merely to bridge sections.
- Avoid nested cards, universal rounded wrappers, default bento/card grids, and over-framed dashboards. Use open layouts, bands, rails, lists, tables, canvases, or one purposeful frame when the spec/product fits. Use stable media crops, aspect ratios, radius, shadow, and spacing. Utility labels, pseudo-system markers, fake metrics, and decorative jargon require comprehension value.
- Use concept-review-only behavior when the user asks for concepts/options or asks to wait: show and iterate concepts, wait for approval, then lock the accepted concept as the active spec. Do not implement during review.
- Accepted concept is immutable production specification: preserve layout, visible copy, hierarchy, container model, styling, imagery, density, and sections. User approval or a concrete recorded blocker is required for any deviation; generic taste never overrides it.

## Pre-code inventory

- Extract a small design system: tokens; content and chrome typography; spacing; icon treatment; component families/variants; responsive rules; container rules; color, surface, border, shadow, radius, elevation, semantic, and motion values.
- Inventory each section/state: concept source and native aspect, priority, readable text, type relationships, spacing, controls, containers, colors, and details needing a new extraction image. Record section order, first viewport, next-section preview, responsive continuation, downstream states, section bridges, allowed component families, and unshown major components as prohibited unless user-requested or functionally necessary.
- List all allowed above-the-fold visible copy from user or accepted concept: hero, nav, label, eyebrow/kicker/pretitle, badge/pill, CTA, and proof strings. Record exact nav, CTA, headings, proof/data labels, and important UI text.
- Inventory every media role, product mockup, table/chart/map/rail/form/HUD, brand mark, and hero treatment: no overlay, color/gradient overlay, edge fade, mask, transparent asset, or matching background. Explicitly lock true white, off-white, cream, gray, dark, or tint; do not reinterpret temperature.
- Create standalone Image Gen edited assets for concept logos, marks, labels, packaging, posters, signs, product renders, and branded backgrounds; generate transparent variants when layering. Games require matching visible art; collision boxes/game state cannot substitute it.
- Record font family/fallback, scales, weights, line heights, tracking, labels, heading/body/caption/control styles, and responsive behavior. Record each icon/glyph/mark with meaning, family, outline/fill, stroke, size, color, container, alignment, spacing, and selected/hover/disabled states.
- For complex apps plan shell, navigation, regions, primitives, feature modules, sample data/state helpers, asset modules, responsive boundaries, and working controls. Keep `App` composition glue, not a copied-markup monolith. If required detail/state/mobile concepts are missing or uncertain, generate matching details; otherwise extend only within the locked system.

## Fidelity implementation

- Build the real usable surface, not a marketing shell for a future app. Follow repository framework, routing, components, styling, state, accessibility, and assets; use focused components, an app shell, shared primitives, feature modules, and tokens for repeated roles.
- Implement each section/state directly from the accepted design. Start with first viewport, compare browser screenshot to its concept, repair visible drift, then proceed; do not defer comparison or flatten section-specific decisions.
- Preserve copy, hierarchy, section order, density, actual palette, type, spacing, radii, borders, shadows, framing, and interaction. Do not add carousels, accordions, pricing, dashboards, forms, tabs, or grids absent from the spec unless user-requested or a recorded functional necessity.
- Do not add above-fold copy, labels, subtitles, category text, eyebrow/kicker/pill/badge, decorative gradient, glow, or overlay absent from the spec. Change heading semantics for accessibility/SEO without inventing visible compensating copy. A concept image with no tint stays untinted; blend with matching asset, cutout, edge fade, mask, or surrounding gradient, never a colored wash. White stays white.
- Deliberately style all control/chrome type: buttons, tabs, inputs, toolbars, sidebars, inspectors, layers, status bars, palettes, export/share, tables, and charts must not inherit browser-default control text.
- Preserve open whitespace, bands, rails, lists, tables, canvases, or full-bleed composition; do not add cards/panels/tiles/grids. UI navigation, text, controls, forms, tables, and labels remain code-native; raster text/branding that belongs inside imagery may remain raster. Never ship a UI screenshot as the app.
- Use Image Gen for central non-icon assets. Request full intended logo/text/mark/label/packaging/signage branding, quote exact required asset text, and use transparent backgrounds for layering. Do not replace branded assets with CSS, mismatched type, approximate labels, placeholder SVG, rough drawing, generic gradient, or stock-like crop. Blend crop, light, edge, shadow, and transparency into the surface.
- Game implementation uses generated sprites/states, tiles, collectibles, hazards, goals, props, and depth layers; tune collision geometry to art. Code-drawn/vector game art must be an intentional deviation or concrete blocker. Use SVG/icon components for directional and disclosure glyphs, not plain text glyphs unless specified.
- Preserve icon metaphor, fill/outline, stroke, corners, optical size, baseline, padding, color, spacing, and state. Prefer the repository icon system; use Lucide only when its style matches; otherwise provide a small production-quality custom SVG with clean viewBox, geometry, joins/caps, balanced negative space, scalable paths, and appropriate `currentColor`/fills. Never trade filled for outline, thick for thin, or specific for generic.
- Make appropriate app controls experiential with local state: selected states, filters/tabs/forms, edits/creation, success, playback/game controls, or simulated generated results. Hero interactivity fits only purposeful product demos; do not force fake chrome into editorial, consumer, venue, food, or background-led work. Motion must clarify hierarchy/state/tangibility, preserve accessibility and reduced motion, and keep semantic markup, stable responsive dimensions, lint/type/test quality.

## Verification and stopping gates

- Browser/IAB first: load, inspect viewport, scroll, and exercise the core workflow. Use Playwright Chromium only when Browser/IAB is unavailable or unreliable; state why. Check desktop/current viewport and mobile. Capture native concept dimensions when practical; otherwise record the blocker and verify current viewport.
- Before handoff, use `view_image` on both accepted concept and current browser screenshot in the same QA pass. Browser inspection alone cannot replace this pair. Ask whether a highly skilled agency would sign off; keep repairing if not.
- Produce a fidelity ledger before final: concept evidence, render evidence, mismatch, fix or justified unresolved deviation. For each relevant section/state use its own concept. Inspect at least five concrete points across copy, layout, type, palette/gradients, asset treatment, spacing/container, responsive behavior, or motion.
- Run an above-the-fold copy diff against allowed copy. Fix or explicitly list added, removed, renamed, or reordered copy; unapproved visible additions fail. Audit all type, including chrome, controls, mobile wraps, computed values when needed; audit icons in nav, buttons, cards, toolbars, sidebars, tables, status, empty, pagination, carousel, and mobile. For canvas/editors separately inspect canvas/document versus chrome, default zoom/pan, and persisted state.
- Verify assets load/frame without obscuring content; core workflow updates real local UI state. Build/test clicks do not replace visual fidelity, native-size comparison, or the written ledger.
- Hard stops: clipped/wrapped primary content; prototype layout; rough seed data; placeholder/unfinished cards; code-drawn game placeholders; invented visible/hero copy; color/gradient drift; white changed to cream/off-white; unapproved hero tint; missing/generic/mismatched icons; poor image blend; debug artifacts; unreadable text/type drift/default control type; overflow; unprofessional responsive collapse; inert controls/fake media; any accepted-spec drift.
- Surface gates: landing/company preserves first viewport, hero, brand/nav/CTA labels, order, next preview, signature imagery; product/SaaS preserves mockups, workflows, feature/proof/brand; dashboards/tools preserve density, sidebar/header/table/tab/timeline/chart/map counts and selection, never table-to-card conversion; canvas/editor preserves zoom/pan, canvas text, chrome, tools, inspector/layers/status/commands/autosave; timeline/planning preserves grid/time axis/rows/events/status/command-center; clone surfaces preserve recognizable skeleton without marketing hero/nav; games verify generated art load/scale/state/collision/movement/action/scoring/hazard/restart; media verifies load/duration/play-pause/seek/progress/frame change; transaction flows verify main path and confirmation.
- Remove temporary QA artifacts unless retention is requested. Final evidence includes concept path, screenshot method, Browser/IAB or Playwright reason, `view_image` pair, native viewport or blocker, five comparisons, copy-diff result, intentional deviations, fixed mismatches, core path, and an explicit faithful-verification statement; directly say when no material mismatch remains.

## Capability coordination

- Coordinate provider-backed behavior with installed capability skills; provider setup never blocks concepting, static UI, or design review that does not use it. Do not write placeholder setup when an installed skill owns it.
- For model-generated output, use `openai-developers:openai-platform-api-key` when available unless user names another provider or opts out. Use its credential flow, never fake keys, placeholder environment variables, or manual key instructions.
